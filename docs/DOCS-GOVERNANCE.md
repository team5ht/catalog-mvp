# Documentation Governance

Этот документ фиксирует правила работы с документацией: что документировать, где хранить, когда обновлять статусы и когда архивировать.

## 1) Types of Documents

- `README.md`
  - Канонический runtime/product контракт проекта.
- `docs/live/*`
  - Актуальные рабочие инструкции и технические регламенты.
- `docs/adr/*`
  - Архитектурные решения (decision log), включая superseded решения.
- `docs/archive/*`
  - Исторические документы, snapshots, аудиты, постфактум-отчеты.

## 2) Document Statuses

Каждый документ в `docs/live/*` и новые документы в `docs/archive/*` должен содержать короткий статус-блок в начале:

```md
Status: Active | Frozen | Superseded | Archived
Owner: <team/person>
Last reviewed: YYYY-MM-DD
Scope: <1 sentence>
```

Интерпретация:

- `Active` - документ поддерживается и должен соответствовать текущему состоянию.
- `Frozen` - документ зафиксирован как финальная версия процесса/релиза и редко меняется.
- `Superseded` - заменен другим документом; должна быть ссылка на замену.
- `Archived` - только историческая ценность, не источник текущих решений.

## 3) Placement Rules (where to put docs)

- В `README.md`:
  - Только краткие и стабильные контракты и инструкции запуска/проверки.
- В `docs/live/*`:
  - Playbook/runbook/checklist, которые нужны в текущей разработке.
- В `docs/adr/*`:
  - Новое архитектурное решение или изменение ключевого решения.
- В `docs/archive/*`:
  - Любой dated audit/snapshot/report после завершения его практической фазы.

## 4) Update Rules (when to edit docs)

При изменении кода обновление документации обязательно, если меняется хотя бы одно:

- публичный контракт (API/route/query/response);
- поведение auth/redirect/security flow;
- процессы запуска, деплоя, тестирования;
- архитектурное решение (требует ADR или update существующего ADR).

Минимальный набор обновлений:

1. `README.md` (если затронут runtime-контракт).
2. Релевантный файл в `docs/live/*`.
3. `docs/INDEX.md` (если добавлен/перенесен документ).

## 5) Archiving Rules (when to move to archive)

Документ переносится в `docs/archive/*`, если выполняется одно из условий:

- появился новый документ-замена с более свежей датой/областью;
- документ описывает состояние на фиксированную дату (snapshot/audit) и не должен эволюционировать как source of truth;
- все action items закрыты и документ нужен только как исторический след;
- документ не обновлялся >30 дней и не используется как текущая инструкция.

Процедура архивирования:

1. Переместить файл в `docs/archive/`.
2. В исходном источнике истины добавить ссылку на архивный документ (если нужно).
3. Обновить пути в `README.md`, `docs/INDEX.md` и других ссылках.
4. Пометить документ как `Archived` или `Superseded` (в шапке).

## 6) Naming Conventions

- Live docs: `kebab-case-topic.md` без даты, если это долгоживущий документ.
- Dated snapshots/audits: `topic-YYYY-MM-DD.md`.
- ADR: `YYYY-MM-DD-topic.md`.

## 7) AI/Agent Context Policy

Чтобы не перегружать контекст:

- По умолчанию читать только:
  - `README.md`
  - `docs/INDEX.md`
  - `docs/live/*`
- `docs/adr/*` читать при архитектурных изменениях.
- `docs/archive/*` читать только для исторического расследования или по прямому запросу.

## 8) Review Cadence

- Еженедельно:
  - проверить `docs/live/*` на актуальность.
- Раз в месяц:
  - переносить устаревшие документы в `docs/archive/*`;
  - синхронизировать `docs/INDEX.md`.
